## 1. Overview

This document defines the API contract for the **feedback service router** that Wonbum implements in Sprint 3.
The router is registered into Colleague A's FastAPI app via `include_router()`, and Colleague B's UI consumes these endpoints.

- **Base URL**: `/api`
- **Content-Type**: `application/json` (all requests and responses)
- **ID format**: UUID v4 (e.g., `"cbca2439-5fa7-4a69-b4d5-57515f2ca8df"`)

---

## 2. Status Lifecycle

```
  ┌─────────┐    TA approve    ┌──────────┐   Instructor publish   ┌───────────┐
  │ pending │ ──────────────→ │ approved │ ─────────────────────→ │ published │
  └─────────┘                  └──────────┘                        └───────────┘
```

| Rule | Description |
|---|---|
| `pending → approved` | TA reviews and approves (requires `ta_id`) |
| `approved → published` | Instructor/Professor publishes to students |
| `pending → published` | **Not allowed** — must go through `approved` first |
| `published → *` | **Not allowed** — `published` is a terminal state |
| `approved → pending` | **Not allowed** — no reverse transitions |

---

## 3. Common Error Format

All error responses use the same format:

```json
{
  "detail": "error message here"
}
```

| HTTP Status | When |
|---|---|
| `200` | Success |
| `404` | Resource not found (draft or submission does not exist) |
| `409` | Conflict (invalid state transition, e.g., approving an already-approved draft) |
| `422` | Validation error (missing required field, auto-generated by FastAPI/Pydantic) |
| `503` | LLM server unavailable (Ollama offline) |

## 4. Endpoints

---

### 4.1 List Feedback Drafts

**`GET /api/submissions/{submission_id}/feedback-drafts`**

Returns all feedback drafts for a given submission.

#### Path Parameters

| Name | Type | Description |
|---|---|---|
| `submission_id` | UUID | Target submission ID |

#### Query Parameters (optional)

| Name | Type | Description |
|---|---|---|
| `status` | string | Filter by status (`pending`, `approved`, `published`) |

#### Response `200 OK`

```json
{
  "submission_id": "cbca2439-5fa7-4a69-b4d5-57515f2ca8df",
  "drafts": [
    {
      "draft_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "model_name": "llama3.2",
      "prompt_version": "v1.0",
      "generated_at": "2026-03-25T14:30:00Z",
      "draft_text_preview": "The code structure is generally well-organized, but edge case handling is insufficient. Specifically, validation for negative inputs is missing, and test case 3 returns 3 instead of expected 5...",
      "confidence": 0.820,
      "status": "pending"
    }
  ],
  "count": 1
}
```

> **Note for Colleague B**: `draft_text_preview` is truncated to the first 200 characters.
> Use Endpoint 4.2 (detail) for the full text.

#### Error `404 Not Found`

```json
{
  "detail": "Submission not found: cbca2439-5fa7-4a69-b4d5-57515f2ca8df"
}
```

### 4.2 Get Feedback Draft Detail

**`GET /api/feedback-drafts/{draft_id}`**

Returns full draft text with evidence list.

#### Path Parameters

| Name | Type | Description |
|---|---|---|
| `draft_id` | UUID | Target draft ID |

#### Response `200 OK`

```json
{
  "draft_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "submission_id": "cbca2439-5fa7-4a69-b4d5-57515f2ca8df",
  "model_name": "llama3.2",
  "prompt_version": "v1.0",
  "generated_at": "2026-03-25T14:30:00Z",
  "draft_text": "The code structure is generally well-organized, but edge case handling is insufficient...\n\n## Strengths\n- Good function decomposition and readability\n- Clear variable naming\n\n## Needs Improvement\n- Add negative input validation in `calculate_total()`\n- Fix off-by-one error in loop (line 42)\n\n## Suggested Score: 75/100",
  "confidence": 0.820,
  "status": "pending",
  "approved_by": null,
  "approved_at": null,
  "published_at": null,
  "evidence": [
    {
      "evidence_id": "e1f2a3b4-c5d6-7890-abcd-ef1234567890",
      "evidence_type": "test_run",
      "pointer": "test_case_003",
      "snippet": "AssertionError: expected 5 but got 3"
    },
    {
      "evidence_id": "f2a3b4c5-d6e7-8901-abcd-ef1234567890",
      "evidence_type": "code_snippet",
      "pointer": "hw1_solution.py#L42",
      "snippet": "for i in range(len(arr)):"
    },
    {
      "evidence_id": "a3b4c5d6-e7f8-9012-abcd-ef1234567890",
      "evidence_type": "rubric_item",
      "pointer": "rubric_item:edge_cases",
      "snippet": "Edge case handling (15 points)"
    }
  ]
}
```

> **Note for Colleague B**: The `evidence` array contains references the LLM used to generate feedback.
> You can render them differently by `evidence_type`:
> - `test_run`: test execution result — red/error highlight
> - `code_snippet`: student code excerpt — code block highlight
> - `rubric_item`: rubric criteria — badge/tag style

#### Error `404 Not Found`

```json
{
  "detail": "Draft not found: a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

### 4.3 Approve Feedback Draft

**`POST /api/feedback-drafts/{draft_id}/approve`**

TA approves a pending draft (`pending` → `approved`).

#### Path Parameters

| Name | Type | Description |
|---|---|---|
| `draft_id` | UUID | Target draft ID |

#### Request Body (**required**)

```json
{
  "ta_id": "t1a2b3c4-d5e6-7890-abcd-ef1234567890"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `ta_id` | UUID | **Yes** | ID of the approving TA |

> **Note**: In the MVP, `ta_id` is passed explicitly in the request body.
> In future sprints, this will be extracted from the LTI 1.3 JWT.

#### Response `200 OK`

```json
{
  "draft_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "approved",
  "approved_by": "t1a2b3c4-d5e6-7890-abcd-ef1234567890",
  "approved_at": "2026-03-25T15:00:00Z"
}
```

#### Error `404 Not Found`

```json
{
  "detail": "Draft not found: a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

#### Error `409 Conflict`

```json
{
  "detail": "Cannot approve: draft status is 'approved', expected 'pending'"
}
```

> **Note for Colleague B**: On 409, display a message like "This feedback has already been approved."

#### Error `422 Validation Error`

```json
{
  "detail": [
    {
      "loc": ["body", "ta_id"],
      "msg": "field required",
      "type": "value_error.missing"
    }
  ]
}
```

### 4.4 Publish Feedback Draft

**`POST /api/feedback-drafts/{draft_id}/publish`**

Instructor/Professor publishes an approved draft to the student (`approved` → `published`).

#### Path Parameters

| Name | Type | Description |
|---|---|---|
| `draft_id` | UUID | Target draft ID |

#### Request Body

None (empty body or `{}`)

> **Note**: In the MVP, no instructor authentication is required.
> Future sprints will add instructor role verification via LTI 1.3.

#### Response `200 OK`

```json
{
  "draft_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "published",
  "published_at": "2026-03-25T16:00:00Z"
}
```

#### Error `404 Not Found`

```json
{
  "detail": "Draft not found: a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

#### Error `409 Conflict`

```json
{
  "detail": "Cannot publish: draft status is 'pending', expected 'approved'"
}
```

> **Note for Colleague B**:
> - A `pending` draft cannot be published directly — it must be approved first.
> - Only enable the Publish button when the draft status is `approved`.

### 4.5 Generate Feedback (Trigger LLM)

**`POST /api/submissions/{submission_id}/generate-feedback`**

Triggers LLM feedback generation for a submission:
1. Reads submission data from DB
2. Assembles a feedback packet (assignment, rubric, code, test results, similarity, history)
3. Calls local Ollama LLM
4. Saves the generated draft to DB with `status = 'pending'`

#### Path Parameters

| Name | Type | Description |
|---|---|---|
| `submission_id` | UUID | Target submission ID |

#### Request Body

None (empty body or `{}`)

#### Response `200 OK`

```json
{
  "draft_id": "newly-generated-uuid-1234-5678-abcdef",
  "submission_id": "cbca2439-5fa7-4a69-b4d5-57515f2ca8df",
  "status": "pending",
  "message": "Feedback draft generated successfully"
}
```

> **IMPORTANT for Colleague B**:
> - This call may take **60–120 seconds** due to LLM inference time on CPU.
> - Show a **loading spinner** and set the HTTP timeout to at least **120 seconds**.
> - A future sprint may convert this to async (`202 Accepted` + polling), but the MVP uses synchronous calls.

#### Error `404 Not Found`

```json
{
  "detail": "Submission not found: cbca2439-5fa7-4a69-b4d5-57515f2ca8df"
}
```

#### Error `503 Service Unavailable`

```json
{
  "detail": "LLM server unavailable: Ollama is not running or unreachable"
}
```

> **Note for Colleague B**: On 503, display a message like "AI server is unavailable. Please try again later."

## 5. Instructions for Colleague A (DB + API)

### 5.1 Run DB Migration

```bash
psql agentic_ta -f db/migrations/001_add_draft_status.sql
```

Verify:
```bash
psql agentic_ta -c "\d llm_feedback_draft"
# Confirm: status, approved_by, approved_at, published_at columns exist
```

### 5.2 Register the Feedback Router

Add the following to your `main.py`:

```python
from app.routers.feedback import router as feedback_router

app.include_router(feedback_router, prefix="/api")
```

### 5.3 Database Dependency Injection (`get_db`)

- If you provide a `get_db()` function, Wonbum will use it.
- If not, Wonbum will implement one in `app/deps.py` (psycopg2-based).

### 5.4 CORS Middleware

For Colleague B's frontend integration, please add CORS middleware:

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000"],  # Colleague B's frontend
    allow_methods=["*"],
    allow_headers=["*"],
)
```

## 6. Instructions for Colleague B (UI)

### 6.1 API Basics

| Item | Value |
|---|---|
| Base URL | `http://localhost:8000/api` (dev environment) |
| Content-Type | `application/json` |
| ID format | UUID v4 string |

### 6.2 UI Component → API Mapping

| UI Component | API Endpoint | Notes |
|---|---|---|
| TA screen: feedback list | `GET /api/submissions/{id}/feedback-drafts` | Uses `draft_text_preview` (200 chars) |
| TA screen: feedback detail/edit | `GET /api/feedback-drafts/{id}` | Full `draft_text` + `evidence` array |
| TA screen: "Generate Feedback" button | `POST /api/submissions/{id}/generate-feedback` | 60-120s latency, show loading spinner |
| TA screen: "Approve" button | `POST /api/feedback-drafts/{id}/approve` | Body requires `ta_id` |
| Instructor screen: "Publish" button | `POST /api/feedback-drafts/{id}/publish` | No body required |
| Student screen: view feedback | `GET /api/submissions/{id}/feedback-drafts?status=published` | Filter by `status=published` |

### 6.3 Error Handling Guide

| HTTP Status | Suggested UI Message |
|---|---|
| `404` | "Feedback not found" |
| `409` | "This feedback has already been processed" (extract specifics from `detail`) |
| `422` | Show form validation errors |
| `503` | "AI server is unavailable. Please try again later." |

### 6.4 Button Enable/Disable Rules by Status

| Draft Status | Generate | Approve | Publish |
|---|---|---|---|
| (no draft) | enabled | disabled | disabled |
| `pending` | disabled | **enabled** | disabled |
| `approved` | disabled | disabled | **enabled** |
| `published` | disabled | disabled | disabled |
