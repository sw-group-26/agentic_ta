# Ingestion & LLM Feedback Draft Module

**Sprint 2 — Wonbum Sohn**
Branch: `feat/wonbum-s2-ingestion-llm`

This document explains the **seed data ingestion pipeline** and the **LLM feedback draft generation** implemented in Sprint 2. It is intended for teammates who need to understand what data lives in the DB and how to build on top of it.

---

## Overview

Sprint 2 delivers a complete **vertical slice**:

```
Seed CSV/ZIP  ──►  PostgreSQL DB + Local Storage  ──►  Feedback Packet  ──►  Ollama LLM  ──►  llm_feedback_draft (DB)
```

1. **Seed ingestion** — loads simulated student submission data (from `make_simul_data/`) into PostgreSQL and a local object storage folder.
2. **Feedback packet assembly** — queries multiple DB tables and assembles a structured context packet for the LLM.
3. **LLM call** — sends the packet to a locally running Ollama model and receives a structured feedback draft.
4. **Draft persistence** — saves the draft text, confidence score, and supporting evidence into the DB.

All 24 unit tests pass. Scripts are idempotent (safe to re-run).

## Detailed Script Reference

### `scripts/ingest_seed_data.py`

Loads the CSV metadata and code artifacts generated by `make_simul_data/` into PostgreSQL and the local object storage folder.

**Usage:**

```bash
python scripts/ingest_seed_data.py --sample demo_case_01
```

**What it writes to the DB (in order):**

| Table | Description |
|-------|-------------|
| `course` | Course record (`CS1301`) |
| `course_offering` | Semester offering (Spring 2026) |
| `student` | 3 student records |
| `enrollment` | Student ↔ offering links |
| `assignment` | HW1 definition |
| `rubric_item` | Rubric criteria from CSV |
| `submission` | 1 row per (student, assignment) — canonical record |
| `submission_version` | 1 row per attempt — includes `attempt_no` |
| `test_run` | Test execution results (score, results JSON path) |
| `similarity_check` | Pairwise similarity scores (AST method) |

**What it writes to storage:**

```
storage/
└── offering/{offering_id}/
    └── assignment/{assignment_id}/
        └── student/{student_id}/
            └── attempt/{attempt_no}/
                ├── solution.py       # student code file
                └── results.json      # test run results
```

**Idempotency:** Re-running the script does NOT create duplicate rows. Each entity is looked up before insertion.

---

### `scripts/generate_feedback_draft.py`

Runs the full feedback pipeline for a single submission: packet assembly → LLM call → DB save.

**Usage:**

```bash
# Use a hardcoded demo submission_id (recommended for first run)
python scripts/generate_feedback_draft.py --demo

# Use a specific submission UUID
python scripts/generate_feedback_draft.py --submission-id <uuid>

# Skip Ollama call — print packet only (for debugging)
python scripts/generate_feedback_draft.py --demo --dry-run
```

**Pipeline steps:**

```
1. build_feedback_packet(submission_id, conn, storage)
      → Queries: submission, assignment, rubric_item, submission_version,
                 test_run, similarity_check
      → Loads: code file from local storage
      → Returns: packet dict (7 sections)

2. generate_feedback(packet)
      → Builds structured prompt from packet
      → POST http://localhost:11434/api/generate
      → Parses JSON response

3. save_draft(submission_id, result, conn)
      → INSERT INTO llm_feedback_draft
      → INSERT INTO llm_evidence (one row per evidence item)
      → Returns draft_id (UUID string)
```

### 6c. DB Tables Written by Sprint 2

#### `llm_feedback_draft`

The primary output table. One row per LLM invocation per submission.

| Column | Type | Description |
|--------|------|-------------|
| `draft_id` | UUID | Primary key |
| `submission_id` | UUID FK | Links to `submission` |
| `model_name` | TEXT | e.g. `"llama3.2"` |
| `prompt_version` | TEXT | e.g. `"v1.0"` |
| `generated_at` | TIMESTAMPTZ | Auto-set by DB |
| `draft_text` | TEXT | Full Korean feedback text |
| `confidence` | NUMERIC(4,3) | 0.000 – 1.000 |

#### `llm_evidence`

Supporting evidence items cited by the LLM. Multiple rows per draft.

| Column | Type | Description |
|--------|------|-------------|
| `evidence_id` | UUID | Primary key |
| `draft_id` | UUID FK | Links to `llm_feedback_draft` |
| `evidence_type` | TEXT | `"failed_test"`, `"similarity_flag"`, `"code_snippet"` |
| `pointer` | TEXT | UUID reference to the source row (test_run, check, version) |
| `snippet` | TEXT | Short excerpt shown in the UI |

#### `submission_version` (updated in Sprint 2)

| Column | Type | Description |
|--------|------|-------------|
| `version_id` | UUID | Primary key |
| `submission_id` | UUID FK | Parent submission |
| `attempt_no` | INTEGER | 1-based attempt index (added Sprint 2) |
| `created_at` | TIMESTAMPTZ | When this version was recorded |

## For DB Teammate

**You provide:** `db/schema.sql` (already applied).

**Sprint 2 schema change:** `submission_version.attempt_no INTEGER NOT NULL` was added. If your local DB was created before Sprint 2, re-apply the schema or run:

```sql
ALTER TABLE submission_version ADD COLUMN IF NOT EXISTS attempt_no INTEGER NOT NULL DEFAULT 1;
ALTER TABLE submission_version ADD CONSTRAINT uq_submission_attempt UNIQUE (submission_id, attempt_no);
```

**Tables you will query in Sprint 3 (FastAPI endpoints):**

```sql
-- List all feedback drafts for a submission (approval queue)
SELECT draft_id, draft_text, confidence, generated_at, model_name, prompt_version
FROM llm_feedback_draft
WHERE submission_id = :submission_id
ORDER BY generated_at DESC;

-- Fetch evidence items for a draft
SELECT evidence_type, pointer, snippet
FROM llm_evidence
WHERE draft_id = :draft_id;

-- List all submission attempts in order
SELECT v.attempt_no, v.created_at, s.submitted_at
FROM submission_version v
JOIN submission s ON s.submission_id = v.submission_id
WHERE v.submission_id = :submission_id
ORDER BY v.attempt_no;
```

**Suggested Sprint 3 schema additions** (to be agreed with team):

```sql
-- Add approval status to feedback drafts
ALTER TABLE llm_feedback_draft
  ADD COLUMN status TEXT NOT NULL DEFAULT 'pending'
  CHECK (status IN ('pending', 'approved', 'published'));
```

## Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| `psycopg2.OperationalError` | DB not running or wrong credentials | Check `DATABASE_URL` in `.env`, confirm PostgreSQL is running |
| `OllamaUnavailableError` | Ollama server is not running | Run `ollama serve` in a separate terminal |
| `OllamaUnavailableError: 5xx` | Model not pulled | Run `ollama pull llama3.2` |
| `KeyError: attempt_no` | Old DB schema without the Sprint 2 column | Re-apply `db/schema.sql` or run the ALTER TABLE in section "For DB Teammate" |
| Duplicate rows on re-run | Should not happen — script is idempotent | If it does, open an issue: `ingest_seed_data.py` uses SELECT-before-INSERT |
| `json.JSONDecodeError` from LLM | Ollama returned non-JSON or truncated output | Increase `OLLAMA_REQUEST_TIMEOUT_SEC` in `.env` (default: 120s) |
